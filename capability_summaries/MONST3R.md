**Paper:** MonST3R: A Simple Approach for Estimating Geometry in the Presence of Motion, ICLR 2025, MONST3R
**Summary:** MonST3R fine-tunes DUSt3R's pairwise pointmap network on dynamic posed-video datasets to directly estimate dense per-timestep geometry from monocular RGB video of dynamic scenes. From pairs of video frames, the network predicts dense 3D pointmaps in the first frame's camera coordinate system; these pairwise estimates are then accumulated into a global dynamic point cloud and per-frame camera poses/intrinsics via a lightweight sliding-window optimization. Downstream outputs include temporally consistent video depth, camera extrinsics, camera intrinsics, and static/dynamic scene segmentation masks.

1. **3D reconstruction:** Yes. MonST3R outputs per-frame dense pointmaps X^t ∈ R^{H×W×3} and accumulates them into a global dynamic point cloud. Fig. 1 shows the "Dynamic Point Cloud & Camera Pose" as primary output; Sec. 3.2 defines per-frame pointmaps; Sec. 3.4 describes the global optimization that produces world-coordinate pointmaps X^t ∈ R^{H×W×3} across all frames.

2. **Dense reconstruction:** Yes. Every pixel is assigned an xyz coordinate in the pointmap — an explicit per-pixel dense 3D representation. This is the core DUSt3R representation adapted for dynamic scenes (Sec. 3.1, 3.2). Sec. 4.4 evaluates "Joint Dense Reconstruction and Pose Estimation."

3. **Reference frame:** Selectable. Each pairwise network call outputs in the first image's camera frame; by choosing which image leads each pair, any frame can act as the local reference, and the global optimisation aligns all pairs into a single consistent world frame — so both world-space and egocentric output are available.

4. **Metric scale:** No. Depth evaluations in Table 2 use per-sequence scale-and-shift or scale-only alignment before scoring. Camera pose is evaluated after Sim(3) alignment (Sec. 4.3). The global alignment objective includes per-pair scale factors σ^e (Eq. 4). No metric-scale output is claimed or demonstrated.

5. **Point tracking:** No. MonST3R does not output persistent point identities or tracked correspondences across frames. Sec. 3.3 explicitly contrasts its approach — using per-pixel 2D-3D correspondences within the same view for PnP — against two-view correspondence methods. Optical flow is consumed from an off-the-shelf model (SEA-RAFT; Wang et al., 2024d) as an input to the static-mask computation and flow loss, not produced as an output. Dynamic objects appear at multiple locations in the point cloud rather than being linked across time (Sec. 3.2).

6. **3D tracks:** No. The output is a time-varying dynamic point cloud where each frame's pointmap is an independent snapshot. No 3D point trajectories with persistent identities across frames are described or output (Sec. 3.2, Sec. 3.4).

7. **2D tracks:** No. Q6=No, so the auto-Yes rule does not apply. No 2D point track output is described in the paper. Optical flow fields are used only as auxiliary signals during optimization (Sec. 3.3, Sec. 3.4), not as outputs.

8. **Occlusion / visibility prediction:** No. The paper derives confident static/dynamic masks S^{t→t'} by comparing camera-induced flow with off-the-shelf optical flow (Sec. 3.3, Eq. 3). These masks identify static versus dynamic regions but are not per-point or per-track occlusion/visibility flags. No such output is described in Fig. 1 or Sec. 3.4.

9. **Multiple views:** No. MonST3R processes monocular video. The pairwise network takes two temporal frames from a single camera stream at a time. The video graph in Sec. 3.4 is defined over sequential frames V = [I^0, …, I^N] from one camera; the sliding temporal window operates on this monocular sequence. No simultaneous multi-camera input is used or claimed.

10. **Global context vs. pairwise vs. fixed-window:** Hybrid. At the network level, inference is pairwise: two frames are processed together to produce pairwise pointmaps. At the global aggregation level, a sliding temporal window of fixed size w is used (Sec. 3.4) to limit compute; pairwise estimates within each window are then jointly optimized. The result is a Hybrid of pairwise feed-forward prediction and fixed-window global optimization.

11. **No GT extrinsics required at inference:** Yes. MonST3R estimates all camera parameters from the input video without ground-truth poses. Sec. 3.3 derives relative pose from predicted pointmaps via PnP/RANSAC; Sec. 3.4 optimizes global poses jointly with the point cloud. Table 4 confirms competitive results against methods that require GT intrinsics as input, and MonST3R requires neither GT extrinsics nor GT intrinsics.

12. **Extrinsics as output:** Yes. Per-frame camera extrinsics P^t = [R^t | T^t] are an explicit output of the global optimization (Sec. 3.4, Eq. 7). Fig. 1 shows "Camera Pose" as a primary output. Relative pose estimation is described in Sec. 3.3 (Eq. 1).

13. **No GT intrinsics required at inference:** Yes. Camera intrinsics are estimated from the pointmap geometry (Sec. 3.3): "we only need to solve for focal length f^t to obtain the camera intrinsics K^t." Table 4 caption explicitly notes that MonST3R achieves competitive results "even without ground truth camera intrinsics."

14. **Intrinsics as output:** Yes. Per-frame focal length f^t and intrinsic matrix K^t are estimated and output (Sec. 3.3). Fig. 1 lists "Camera Intrinsics" as a primary output. The global optimization reparameterizes the scene via camera pose, intrinsics, and per-frame depth (Sec. 3.4). Sec. 4.5 notes that varying camera intrinsics are theoretically supported but require careful hyperparameter tuning in practice.

15. **Moving cameras:** Yes. Handling moving cameras is a primary use case. Sec. 3.3 estimates relative camera pose between each pair of frames; Sec. 3.4 optimizes a full per-frame camera trajectory. Evaluations include camera pose estimation on Sintel, TUM-dynamics, and ScanNet with free camera motion (Sec. 4.3, Table 4).

16. **Dynamic content:** Yes. This is the core contribution of the paper. MonST3R is explicitly designed to estimate geometry from scenes where objects move and deform over time. The abstract states: "directly estimates per-timestep geometry from dynamic scenes." Fine-tuning datasets include dynamic articulated objects (Table 1), and the key design choices — dynamic pointmap representation, confident static region masking, flow projection loss — all address dynamic content (Sec. 3.2, 3.3, 3.4).

17. **Feed-forward inference:** Partial. The pairwise pointmap prediction network is entirely feed-forward. However, the full multi-frame reconstruction pipeline includes a per-scene global optimization: ~300 iterations of Adam at learning rate 0.01, taking approximately 1 minute for a 60-frame video on a single RTX 6000 GPU (Sec. 4.1). The paper itself acknowledges this, describing the system as "predominantly feed-forward" (Abstract, Fig. 1 caption). The global optimization constitutes a test-time per-scene optimization step, precluding a verdict of fully feed-forward.

**Notes:** Input modality is monocular RGB video; no depth sensor, stereo, or multiple cameras are used. The model is fine-tuned from DUSt3R (ViT-Base encoder frozen; decoder and DPT heads fine-tuned) using supervised datasets with camera pose and depth annotations — primarily synthetic (PointOdyssey, TartanAir, Spring) plus real-world driving (Waymo with LiDAR depth). All depth evaluations use per-sequence scale or scale-and-shift alignment, and pose evaluations use Sim(3) alignment; downstream users should not assume metric-scale output. The static/dynamic mask output (Fig. 1) is derived from comparing camera-induced flow to off-the-shelf optical flow (SEA-RAFT), so the mask quality depends partly on that external component. The global optimization window size w trades off reconstruction quality against memory and runtime (discussed in Sec. 4.1 and ablations). MonST3R struggles with out-of-distribution inputs such as open fields and is vulnerable to long-term occlusion due to the sliding-window design (Sec. 4.5 Discussions).
