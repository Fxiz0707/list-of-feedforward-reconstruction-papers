**Paper:** MASt3R-SLAM: Real-Time Dense SLAM with 3D Reconstruction Priors, arXiv 2024 (arXiv:2412.12392), MASt3R-SLAM
**Summary:** MASt3R-SLAM is a real-time monocular dense SLAM pipeline built around the off-the-shelf two-view MASt3R reconstruction and matching prior. It incrementally tracks camera motion via pairwise MASt3R inference, fuses dense per-pixel pointmaps into canonical keyframe maps, detects loop closures via image retrieval, and runs second-order Gauss-Newton global optimisation to produce globally consistent camera poses and dense scene geometry at ~15 FPS. The full method is not feed-forward: it includes iterative matching, Gauss-Newton tracking, running pointmap fusion, loop-closure graph construction, and backend optimisation per scene.

1. **3D reconstruction:** Yes. The system explicitly outputs dense globally consistent 3D geometry. MASt3R produces dense pointmaps X^i, X^j ∈ R^{H×W×3} per image pair; these are fused into canonical keyframe pointmaps X̃^k and jointly optimised in the backend. Dense point cloud outputs are evaluated against ground-truth structure (Sec. 3.3–3.5, pp. 4–5; Sec. 4.2, pp. 6–7; Figs. 1, 4, 6, 7, 9, 10).

2. **Dense reconstruction:** Yes. MASt3R-SLAM is explicitly a dense SLAM system. MASt3R outputs per-pixel pointmaps of shape H×W×3 with per-pixel confidence maps. Local fusion maintains dense canonical pointmaps and the supplementary notes that estimated points are not filtered because "downstream applications benefit from per-pixel dense prediction" (Sec. 3.1, p. 3; Sec. 3.3, pp. 4–5; Sec. 11.2, p. 14).

3. **Reference frame:** Fixed world. The SLAM backend maintains a fixed gauge world frame (T_{WC_i}); while the pairwise network can use any image as the first view, the system-level world frame is locked to the SLAM initialisation and not user-selectable.

4. **Metric scale:** No. The system is monocular and treats scale as a free variable: poses are optimised in Sim(3) with an explicit scale parameter s ∈ R, and the paper acknowledges that "scale is often a large source of inconsistency across predictions" (Sec. 3.1, p. 3). Trajectory evaluation uses scaled alignment for all monocular systems: "Since all systems are monocular, we perform scaled trajectory alignment" (Sec. 4.1, p. 6).

5. **Point tracking:** Yes. Pointmap matching (Sec. 3.2, pp. 3–4) establishes dense pairwise pixel correspondences m_{i,j} = M(X^i, X^j, D^i, D^j) between frames via iterative projective matching and feature refinement. These correspondences are stored as graph edges and reused for tracking, loop closure, and backend optimisation. They are pairwise correspondences rather than persistent long-range point identities.

6. **3D tracks:** No. MASt3R-SLAM does not define or output persistent 3D point trajectories over time. The backend optimises world-frame poses and canonical keyframe pointmaps using pairwise graph edges, but points are fused into single canonical maps rather than tracked as labelled trajectories across many frames (Secs. 3.3–3.5, pp. 4–5).

7. **2D tracks:** Partial. The method computes pairwise 2D pixel correspondences between frames and keyframes as part of matching (Sec. 3.2, pp. 3–4), and tracking reuses previous-frame pixel matches to warm-start convergence. However, long-range 2D point tracks are not a system output product; correspondences are used only internally for pose and geometry estimation.

8. **Occlusion / visibility prediction:** No. The pipeline invalidates matches with large 3D distances "to handle occlusions and outliers" and uses MASt3R confidence weights (q_{min} threshold, Eq. 5, Sec. 3.2, p. 4), but these are filtering heuristics rather than explicit per-point or per-track occlusion/visibility flag outputs.

9. **Multiple views:** No. MASt3R-SLAM is a monocular SLAM system processing a single video stream. MASt3R is invoked pairwise on selected frame/keyframe or loop-closure pairs, not on simultaneous multi-camera inputs. All evaluations are on monocular RGB sequences (Abstract, p. 1; Fig. 3; Sec. 4, p. 6).

10. **Global context vs. pairwise vs. fixed-window:** Hybrid. At the network level, MASt3R is invoked pairwise (current frame vs. current keyframe, or loop-closure pairs). The frontend fuses pointmaps locally via running weighted average. The backend constructs a full pose graph with sequential and loop-closure edges and performs global Gauss-Newton optimisation over all keyframe poses and geometry simultaneously (Secs. 3.2–3.5, pp. 3–5).

11. **No GT extrinsics required at inference:** Yes. The system estimates all camera poses from monocular RGB input with no ground-truth trajectory provided at inference. Ground-truth poses are used only for evaluation metric computation and scaled trajectory alignment (Abstract, p. 1; Sec. 3.3, pp. 4–5; Sec. 4.1, p. 6).

12. **Extrinsics as output:** Yes. Per-frame relative poses T_{kf} are estimated during tracking; the backend outputs world-frame keyframe poses T_{WC_i}. Absolute trajectory error (ATE) is the primary evaluation metric across all benchmarks (Sec. 3.3, pp. 4–5; Sec. 3.5, p. 5; Sec. 4.1, pp. 6–7; Tables 1–3, 9).

13. **No GT intrinsics required at inference:** Yes. A central claim is operation without known calibration or a fixed parametric camera model. The system assumes only a generic central camera (all rays pass through a unique centre) and is evaluated in an "Ours*" uncalibrated mode. Known calibration is optional and handled as a separate variant (Abstract, p. 1; Sec. 3.1, p. 3; Sec. 3.7, p. 6).

14. **Intrinsics as output:** No. The uncalibrated system derives normalised rays ψ(X^i) from each predicted pointmap, implicitly defining a generic central camera model per frame, but it does not output standard parametric intrinsics (focal length, principal point, distortion coefficients). In the calibrated mode, intrinsics are provided as input, not estimated as output (Sec. 3.1, p. 3; Sec. 3.7, p. 6).

15. **Moving cameras:** Yes. MASt3R-SLAM is a monocular SLAM system explicitly designed to estimate camera motion. It is evaluated on moving-camera RGB datasets including TUM RGB-D, 7-Scenes, ETH3D-SLAM, and EuRoC, including sequences with fast camera motion (Abstract, p. 1; Sec. 3.3, pp. 4–5; Sec. 4.1, pp. 6–7).

16. **Dynamic content:** No. The method targets dense SLAM for a static scene observed by a moving camera. No component models or segments independently moving or non-rigid scene content. Confidence-based match invalidation provides some implicit robustness to outlier points but dynamic scene handling is not a stated capability or evaluated property (Secs. 3.2–3.5, pp. 3–5; Sec. 4, pp. 6–7).

17. **Feed-forward inference:** No. The underlying MASt3R prior is a feed-forward two-view network, but the full MASt3R-SLAM inference pipeline includes iterative projective matching (10 LM iterations per pixel, Sec. 3.2, p. 4), Gauss-Newton/IRLS tracking (Sec. 3.3, p. 4), running pointmap fusion, loop-closure graph construction (Sec. 3.4, p. 5), and up to 10 Gauss-Newton backend iterations per new keyframe (Sec. 3.5, p. 5; Sec. 10, p. 13). All of these involve per-scene optimisation.

**Notes:** Input modality is monocular RGB video; no depth sensor, IMU, or stereo input is used or required. The system operates in two modes: uncalibrated (Ours*), which uses a generic central-camera ray representation and requires no knowledge of intrinsics, and calibrated (Ours), which uses known intrinsics to constrain pointmap backprojection and switch to pixel-space residuals. The dense geometry is produced by a SLAM pipeline wrapped around an off-the-shelf MASt3R prior; the network itself is not fine-tuned or retrained as part of this work. Scale is treated as monocular/up-to-scale with Sim(3) pose optimisation; all trajectory comparisons use scaled alignment. The system runs at ~15 FPS on a desktop GPU (RTX 4090) by subsampling every 2 frames. A key limitation noted in Sec. 5 is that global optimisation refines poses but does not currently refine all geometry variables; globally consistent pointmap refinement is left for future work. MASt3R is trained on pinhole images and geometry quality degrades with high distortion.
