**Paper:** MapAnything: Universal Feed-Forward Metric 3D Reconstruction, arXiv 2025, MapAnything
**Summary:** MapAnything is a transformer-based feed-forward model that takes one or more RGB images plus optional geometric inputs such as ray directions or intrinsics, poses, depth, partial reconstructions, and metric scale cues. It directly predicts a factored metric 3D scene representation: dense per-view ray maps and ray depths, camera poses in the first-view frame, and a global metric scale factor, enabling image-only uncalibrated SfM as well as calibrated MVS, depth completion, and related tasks in one inference pass.

1. **3D reconstruction:** Yes. MapAnything produces a dense metric 3D reconstruction: Section 3 defines outputs as local ray directions, ray depths, poses, and a global metric scale factor, then converts them into local pointmaps and metric world-frame pointmaps for all input views in the frame of the first image (Sec. 3, pp. 3-4). The abstract also states that it directly regresses metric 3D scene geometry and cameras (p. 1).

2. **Dense reconstruction:** Yes. The default representation is dense pixelwise geometry rather than sparse keypoints: the DPT head decodes each view into dense ray directions, ray depths, masks, and confidence maps, and these are used to form per-pixel pointmaps (Sec. 3.2, p. 4). The main experiments are dense multi-view reconstruction and dense depth or pointmap evaluation over two to one hundred views (Sec. 4, pp. 6-7; Fig. 4, p. 7).

3. **World-space output:** Yes. MapAnything outputs geometry in a shared scene frame by predicting each image pose in the frame of image 1 and using those poses plus the metric scale factor to compute world-frame metric pointmaps for all views (Sec. 3, p. 4).

4. **Camera-space output (egocentric):** Yes. MapAnything also has camera-local geometry: the predicted local ray directions and ray depths form local pointmaps in each view's camera coordinate system before being transformed into the first-view world frame (Sec. 3, p. 4; Sec. 3.2, p. 4).

5. **Flexible camera frame:** Partial. The global output frame is flexible only through the ordering of the input views: the paper fixes the reconstruction frame to image 1 and adds a reference-view embedding to the first view's features (Sec. 3, p. 4; Fig. 2, p. 3). A user can choose which input camera is first, but the method does not describe selecting an arbitrary external coordinate frame.

6. **Metric scale:** Yes. Metric scale is a core output: the model predicts a global metric scaling factor and applies it to the up-to-scale world-frame pointmaps to produce final metric 3D reconstructions (Sec. 3, p. 4; Sec. 3.2, p. 4). The paper evaluates metric scale and metric depth directly, while noting that metric scale estimation can be sub-optimal on some image-only benchmarks such as ScanNet (Sec. 4, p. 7; Tables 2 and 4, pp. 7-8).

7. **Point tracking:** No. MapAnything aligns dense per-view pointmaps into a common frame, but it does not output explicit point identities, matches, correspondences, or tracks across views. Its predicted quantities are ray directions, ray depths, masks, confidence maps, poses, and metric scale (Sec. 3.2, p. 4).

8. **3D tracks:** No. The method does not output 3D point trajectories. The limitations explicitly state that the current scene parameterization does not capture dynamic motion or scene flow (Sec. 5, p. 8).

9. **2D tracks:** No. The paper does not describe a 2D tracking head or evaluate 2D point tracks. Although reconstructed geometry could be projected into images, the model outputs factored geometry, poses, masks, confidences, and scale rather than 2D track identities (Sec. 3.2, p. 4; Sec. 4, pp. 6-8).

10. **Occlusion / visibility prediction:** No. The dense head predicts per-pixel masks M_i (binary labels for non-ambiguous vs. ambiguous depth regions such as sky and reflections) and per-pixel confidence maps C_i for the world-frame pointmap; these serve to down-weight unreliable pixels during training and are not designed as or evaluated as per-point occlusion or visibility flags in the tracking sense (Sec. 3.2, p. 4; Sec. 3.3, p. 5). No occlusion or visibility benchmark is reported (Sec. 4, pp. 6-8).

11. **Multiple views:** Yes, arbitrary number. MapAnything takes a set of `N` RGB images/views with optional per-view geometric inputs, processes the `N` view patch tokens through a multi-view transformer, and outputs an `N`-view reconstruction; this is evaluated as dense multi-view reconstruction from 2 to 100 input views, not merely a monocular video sequence (Sec. 3, pp. 3-4; Sec. 4/Fig. 4, pp. 6-7). The model supports any N and uses covisibility-based sampling to handle large unordered image collections.
12. **Global context vs. pairwise vs. fixed-window:** Global. The N view patch tokens and a scale token are processed jointly by an alternating-attention multi-view transformer before per-view decoding, so inference uses global multi-view context rather than pairwise stitching or a fixed local window (Sec. 3.1, p. 4; Fig. 2, p. 3). The paper notes memory limits from the one-to-one mapping between input pixels and output scene representation, but this does not change the global transformer aggregation design (Sec. 5, p. 8).

13. **No GT extrinsics required at inference:** Yes. Ground-truth extrinsics are not required at inference: image-only uncalibrated structure-from-motion is a core use case, and poses are optional geometric inputs rather than mandatory inputs (Abstract, p. 1; Sec. 3, p. 3). When poses are not provided, the model predicts them (Sec. 3, p. 4; Sec. 3.2, p. 4).

14. **Extrinsics as output:** Yes. Extrinsics are direct model outputs: MapAnything predicts per-view poses in the frame of image 1, represented as quaternions and up-to-scale translations, with metric scale handled by the global scale factor (Sec. 3, p. 4; Sec. 3.2, p. 4).

15. **No GT intrinsics required at inference:** Yes. Ground-truth intrinsics are not required at inference because image-only settings are evaluated, and the model predicts local ray directions corresponding to a generic central camera calibration (Sec. 3, pp. 3-4; Table 2, p. 7). The paper also reports single-image calibration results without requiring calibration as input (Sec. 4, p. 7; Table 3, p. 7).

16. **Intrinsics as output:** Yes. MapAnything outputs camera calibration as dense local ray directions for a generic central camera model, not as a conventional pinhole intrinsics matrix (Sec. 3, pp. 3-4; Sec. 3.2, p. 4). The paper evaluates this output through ray-direction error and single-image calibration benchmarks (Fig. 4, p. 7; Table 3, p. 7).

17. **Moving cameras:** Yes. The method handles moving cameras by predicting camera poses across multiple input views and evaluating SfM, pose, and trajectory metrics on multi-view scene samples (Abstract, p. 1; Sec. 3, p. 4; Sec. 4, pp. 6-7).

18. **Dynamic content:** No. MapAnything does not model dynamic content as motion or scene flow; the limitations state that the current scene parameterization does not capture dynamic motion or scene flow (Sec. 5, p. 8). Dynamic scenes are framed as a future extension rather than a supported reconstruction capability (Sec. 6, p. 8).

19. **Feed-forward inference:** Yes. MapAnything is explicitly feed-forward: it directly regresses metric geometry and cameras in a single pass without bundle adjustment or costly post-processing (Abstract, p. 1; Sec. 3.2, p. 4). The conclusion reiterates that the model regresses metric 3D geometry and camera poses from flexible inputs in a single pass (Sec. 6, p. 8).

**Notes:** MapAnything's visual input is RGB images, with optional conditioning on ray directions or intrinsics, poses, depth maps, sparse depth, partial reconstructions, and metric scale cues. It is trained end-to-end over thirteen datasets with flexible geometric-input augmentation, but supervision varies by dataset and the released model variants differ by training data and license. Its outputs are dense metric geometry and cameras, not explicit tracks, correspondences, occlusion labels, or dynamic scene flow; the shared coordinate frame is anchored to the first input view.
