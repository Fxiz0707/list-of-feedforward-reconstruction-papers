**Paper:** VGGT-SLAM: Dense RGB SLAM Optimized on the SL(4) Manifold, arXiv preprint (May 2025), VGGT-SLAM
**Summary:** VGGT-SLAM is a dense monocular RGB SLAM system that incrementally builds large-scale scene reconstructions by creating local submaps using the feed-forward model VGGT, then globally aligning them via factor graph optimization on the SL(4) Lie group manifold to address projective ambiguity inherent in uncalibrated camera reconstructions. It operates without known camera intrinsics or extrinsics and requires no additional training beyond what VGGT provides. The full system is not a single feed-forward pass: it includes keyframe selection, SALAD-based loop retrieval, RANSAC homography estimation, and iterative Levenberg-Marquardt backend optimization per scene.

1. **3D reconstruction:** Yes. VGGT-SLAM produces dense point cloud maps. Each submap S contains a dense point cloud X^S built by inverse-projecting VGGT's per-frame depth maps D using camera projection matrices from the camera head; these submaps are globally aligned into a single consistent scene-wide dense map. Dense reconstruction is evaluated quantitatively with accuracy, completion, and Chamfer distance metrics (Sec. 3–4, pp. 3–4; Sec. 5.3, Table 3, p. 8).

2. **Dense reconstruction:** Yes. The system uses VGGT's dense depth maps D = {D_1, ..., D_w}, with one depth value per pixel per frame, to construct submap point clouds. Components of M, D, and C share the same width and height (i.e., depth is image-sized). Low-confidence points are pruned via confidence threshold tau_conf but the representation is fundamentally per-pixel. Dense reconstruction metrics are reported in Sec. 5.3, Table 3 (p. 8).

3. **World-space output:** Yes. The central contribution of VGGT-SLAM is nonlinear factor graph optimization on the SL(4) manifold (Sec. 4.4, p. 6) that estimates absolute homographies H_i transforming all submaps into a single globally consistent reference frame. The output is a unified world-space dense map and globally consistent camera trajectory, confirmed by dense map and pose visualization across 22 submaps in Fig. 2 (p. 8).

4. **Camera-space output (egocentric):** Partial. VGGT (the integrated backbone) outputs per-frame point clouds "defined with respect to the coordinate frame of the first camera" in each submap window (Sec. 3, p. 4). These submap-local, first-camera-frame point clouds exist as intermediate representations used during homography estimation (Sec. 4.2). However, VGGT-SLAM's primary exposed output is a world-space aligned map; the per-submap egocentric representations are internal to the submap generation step.

5. **Flexible camera frame:** N/A. Q4 is Partial/limited. Within each submap the reference frame is fixed to the first camera of the submap window by VGGT's design — it is not freely choosable (Sec. 3, p. 4).

6. **Metric scale:** No. The paper explicitly revisits the Projective Reconstruction Theorem to argue that, with uncalibrated monocular cameras, the reconstruction is determined only up to a 15-DOF projective transformation (homography) belonging to SL(4). VGGT may approximate metric scale from learned priors in favorable conditions, but "in the most general case when estimates of scene priors are unreliable, the reconstruction differs by a projective transformation" requiring SL(4) rectification (Sec. 4.2, pp. 4–5; Abstract, p. 1; Conclusion, p. 9).

7. **Point tracking:** No. The paper explicitly states "we do not need to estimate correspondences between frames" (Sec. 2, p. 3). Homographies between overlapping submaps are computed directly from shared frames included by construction in consecutive submaps, bypassing point matching. VGGT does output dense features for point tracking internally (Sec. 3), but VGGT-SLAM does not use or expose point correspondences or tracking as an output. The "We do not run the 3D point DPT head" statement (Sec. 3, p. 4) further confirms tracking is unused.

8. **3D tracks:** No. The system produces a static aligned point cloud map and camera poses. No 3D point trajectories over time are output or evaluated; Sec. 4 describes submap generation, homography estimation, loop closures, and factor graph optimization without any concept of persistent point identity across frames.

9. **2D tracks:** No. No 2D point tracking output is described or used in VGGT-SLAM. Lucas-Kanade is used internally only as a keyframe-selection disparity signal (Sec. 4.1, p. 4), not as a tracked-point output. The auto-Yes condition (Q8=Yes and Q14=Yes) is not met since Q8=No.

10. **Occlusion / visibility prediction:** No. VGGT's confidence maps C are used to prune low-confidence depth estimates below threshold tau_conf (Sec. 3, p. 4), but these are reconstruction confidence scores, not per-point or per-track occlusion/visibility flags. No occlusion prediction output is described or evaluated anywhere in the paper.

11. **Multiple views:** No. VGGT-SLAM operates exclusively with a single monocular RGB camera. The abstract and contributions explicitly state the system "operates entirely with monocular RGB cameras" (p. 2). Each submap processes a temporal window of frames from one camera; no simultaneous multi-camera or multi-view input is involved.

12. **Global context vs. pairwise vs. fixed-window:** Hybrid. At the local level, VGGT processes a fixed window of w sequential frames per submap (Fixed-window), bounded by GPU memory (tested at w = 8, 16, 32; up to ~60 frames on RTX 4090 in principle). At the global level, VGGT-SLAM estimates pairwise relative homographies between sequential and loop-closure submaps, then performs global factor graph optimization incorporating all constraints to achieve full-sequence consistency (Global). The system is therefore Hybrid: Fixed-window local inference + Global pose graph backend (Sec. 1, p. 2; Sec. 4.1–4.4, pp. 4–6).

13. **No GT extrinsics required at inference:** Yes. No ground-truth camera extrinsics are provided or required. The contributions explicitly state the system "does not require known camera intrinsics or consistent calibration across frames" (p. 2). All camera poses are estimated from VGGT's feed-forward predictions and refined through the SL(4) factor graph. Tables 1 and 2 evaluate VGGT-SLAM in the uncalibrated setting throughout (Sec. 5.1–5.2, pp. 6–7).

14. **Extrinsics as output:** Yes. Camera poses are a primary output. VGGT estimates per-frame camera poses with respect to the first camera of each submap; VGGT-SLAM transforms these into globally consistent absolute camera poses via homography correction: "P_i = H_j^{i^{-1}} P_j ... We can then decompose P to recover the camera pose" (Sec. 4.2, p. 5). Pose estimation is quantitatively evaluated using ATE on 7-Scenes and TUM RGB-D (Sec. 5.2, Tables 1–2, pp. 7–8).

15. **No GT intrinsics required at inference:** Yes. Explicitly stated in contributions: "does not require known camera intrinsics or consistent calibration across frames" (p. 2). Tables 1 and 2 distinguish calibrated baselines (gray rows) from uncalibrated ones; all VGGT-SLAM configurations are evaluated in the uncalibrated setting (Sec. 5.1, p. 6).

16. **Intrinsics as output:** Yes. VGGT's camera head estimates both intrinsics and camera poses as part of its feed-forward output: "The output tokens can then be passed to a camera head to estimate intrinsics and camera poses" (Sec. 3, p. 3). These estimated intrinsics are used in VGGT-SLAM to construct camera projection matrices P for homography estimation and decomposition: "P ∈ R^{4×4} is the camera matrix created from the poses and intrinsic estimates from VGGT" (Sec. 4.2, p. 5). Intrinsics are thus output by the integrated model, though VGGT-SLAM does not separately benchmark intrinsic accuracy.

17. **Moving cameras:** Yes. The system is designed entirely for monocular video SLAM with a continuously moving camera through static environments. It handles long sequences (hundreds to thousands of frames), incremental keyframe selection based on inter-frame disparity, and loop closure detection — all predicated on camera motion. Evaluated on 7-Scenes and TUM RGB-D, standard moving-camera SLAM benchmarks (Sec. 4.1, p. 4; Sec. 5.1–5.2, pp. 6–7).

18. **Dynamic content:** No. The paper does not address or claim any capability for handling moving objects or non-rigid scene content. All experiments are conducted on standard indoor static-scene benchmarks (7-Scenes, TUM RGB-D). The limitations section discusses planar degeneracy, outlier sensitivity, and SL(4) drift — not dynamic content (Sec. 6, p. 9).

19. **Feed-forward inference:** No. While VGGT itself is a feed-forward model within each submap window, VGGT-SLAM as a whole performs per-scene iterative optimization. The backend (Sec. 4.4, p. 6) solves a nonlinear factor graph via Levenberg-Marquardt: "we iteratively compute state increments by solving a linearized least squares problem... the poses are updated on the Lie group as H <- H Exp(delta)." Additionally, the system runs RANSAC for each submap pair and SALAD-based image retrieval for loop closures. These components constitute per-scene test-time processing that is not a single forward pass.

**Notes:**
- **Input modality:** Monocular RGB video only. No depth sensor, no stereo, no known calibration required.
- **Training regime:** VGGT-SLAM requires no additional training. It uses VGGT's pre-trained weights as a fixed feed-forward backbone; the SL(4) factor graph optimization is a classical/unsupervised inference procedure applied per scene.
- **Scale ambiguity:** Projective ambiguity (up to 15-DOF homography in SL(4)) is the central technical challenge. True metric scale is not guaranteed; the system addresses global alignment consistency, not metric correctness per se.
- **Loop closure:** Uses SALAD [26] image descriptors for image retrieval; detected loop frames are passed to VGGT to estimate relative homographies added as loop closure constraints in the factor graph (Sec. 4.3, p. 5).
- **Submap window size:** Parameter w (tested at w = 8, 16, 32) controls the number of new keyframes per submap, balancing reconstruction quality vs. GPU memory; a single VGGT submap is limited to ~60 frames on an RTX 4090 with 24 GB VRAM (Sec. 1, p. 2).
- **Limitation — planar scenes:** Full 15-DOF homography estimation is degenerate for planar scenes (e.g., TUM floor sequence), leading to non-unique/unstable solutions and poor overall reconstruction (Sec. 5.2, Sec. 6, pp. 7, 9).
- **Limitation — outlier sensitivity:** 5-point RANSAC for homography estimation is sensitive to high outlier ratios and to adversarially coherent outliers arising from VGGT's local geometric consistency (Sec. 6, p. 9).
- **Limitation — scale/drift:** SL(4) drift includes not only translation/rotation/scale drift but also perspective drift, making loop closure more complex than in classical SLAM (Sec. 6, p. 9).
- **Benchmarks:** Evaluated on 7-Scenes (pose + dense reconstruction) and TUM RGB-D (pose only) in uncalibrated mode against DROID-SLAM and MAStR3-SLAM; also compared to calibrated SLAM methods for reference.
- **No point tracking output:** Despite VGGT's dense feature outputs supporting point tracking, VGGT-SLAM explicitly avoids computing correspondences between frames (Sec. 2, p. 3) and does not run VGGT's 3D point DPT head (Sec. 3, p. 4).
