**Paper:** DELTA: Dense Efficient Long-range 3D Tracking for Any Video, ICLR 2025, DELTA
**Summary:** DELTA is a feed-forward dense 3D tracking model that takes monocular RGB-D video as input — where depth maps are supplied by an off-the-shelf monocular depth estimator — and tracks every pixel through the video, predicting per-frame 2D pixel coordinates, log-depth, and per-point visibility in a single forward pass. The core output is dense per-pixel 3D trajectories expressed in each target frame's own camera coordinate system; world-space reconstruction and camera pose estimation appear only in an appendix optimization pipeline built on top of DELTA tracks, not in the core model.

1. **3D reconstruction:** Yes. DELTA produces dense, occlusion-aware 3D trajectories `P in R^{T x H x W x 4}`, where each entry encodes target-frame 2D pixel coordinates, depth, and visibility for every source pixel (Sec. 3, p. 3). Appendix A.6 (p. 20) further states that `P(0,nu)` represents the 3D positions of all initial-frame pixels in the destination frame's camera coordinate system, and Fig. 9 visualizes these as dense 3D point clouds.

2. **Dense reconstruction:** Yes. Dense tracking is the paper's core contribution: DELTA tracks every pixel (H x W tracks) from a monocular video (Abstract, p. 1; Introduction, p. 2). Sparse mode is discussed only as an appendix adaptation for sparse benchmark evaluation and is not the default method (Appendix A.3, p. 17).

3. **Reference frame:** Egocentric. DELTA outputs 3D positions in the camera coordinate system of the current processing window's reference frame; a globally consistent world frame is not a model output — world-space alignment in Appendix A.6 requires a separate per-scene optimisation step.

4. **Metric scale:** No. DELTA predicts log-depth ratios `log(d_t/d_1)`, explicitly designed to be scale-invariant and decoupled from the arbitrary scale of input depth maps (Sec. 3.4, p. 7). At inference, 3D scale is entirely inherited from the off-the-shelf depth estimator supplying the input depth. The conclusion acknowledges that "3D tracking performance is closely tied to the accuracy and temporal consistency of the off-the-shelf monocular depth estimation" (Sec. 5, p. 10).

5. **Point tracking:** Yes. Establishing correspondences for every pixel is the method's primary function. The transformer iteratively updates per-pixel track tokens with positions, appearance features, correlation features, and depth correlation to predict trajectories over hundreds of frames (Introduction, p. 2; Sec. 3.1–3.2, pp. 3–6).

6. **3D tracks:** Yes. DELTA outputs 3D tracks as per-source-pixel trajectories `(u_t, v_t, d_t, o_t)` across frames (Sec. 3, p. 3). These are evaluated on Kubric3D (Table 3), LSFOdyssey (Table 4), and TAP-Vid3D (Tables 5, 7) with APD_{3D}, AJ, and OA metrics (Sec. 4.2, pp. 8–9).

7. **2D tracks:** Yes. DELTA directly predicts 2D target-frame pixel coordinates `(u_t, v_t)` as a native model output (Sec. 3, p. 3). It is explicitly evaluated on long-range 2D optical flow (CVO, Table 2) and 2D point tracking (TAP-Vid, Table 8) benchmarks (Sec. 4.2, pp. 8–9; Appendix A.3, p. 17).

8. **Occlusion / visibility prediction:** Yes. Each tracked point carries a binary visibility flag `o_t in {0,1}`, trained with a binary cross-entropy visibility loss weighted by lambda_{visib} (Sec. 3, p. 3; Sec. 4.1, p. 8). Evaluated via IoU (CVO, Table 2) and OA (3D tracking, Tables 3–5).

9. **Multiple views:** No. The input is a single monocular RGB-D video stream. No simultaneous multi-camera or multi-view input is described or evaluated (Sec. 3, p. 3).

10. **Global context vs. pairwise vs. fixed-window:** Fixed-window. The transformer processes a fixed window of T frames at a time; the conclusion explicitly identifies "relatively short temporal processing windows" as a limitation and recommends fewer than a few hundred frames (Sec. 5, p. 10). Within each window, DELTA uses global spatial attention (via anchor tracks) plus local spatial attention and temporal self-attention across all T frames — so context within the window is global, but the aggregation scheme is fixed-window overall (Sec. 3.1–3.2, pp. 4–6).

11. **No GT extrinsics required at inference:** Yes. Ground-truth camera extrinsics are not part of the input specification. The method requires only an RGB-D video (Sec. 3, p. 3); camera poses are estimated only in the optional appendix application pipeline (Appendix A.6, pp. 20–21).

12. **Extrinsics as output:** Pipeline. The core feedforward DELTA model outputs dense pixel tracks `(u_t, v_t, d_t, o_t)`, not camera extrinsics (Sec. 3, p. 3). Per-keyframe extrinsics are produced only by the appendix global-alignment optimization pipeline that takes DELTA tracks as input and runs gradient descent to estimate camera parameters (Appendix A.6, pp. 20–21; Table 10, p. 21).

13. **No GT intrinsics required at inference:** Yes. Ground-truth camera intrinsics are not listed as required input for the core tracker; the problem setup specifies RGB-D video as input (Sec. 3, p. 3). The appendix application optimizes per-keyframe intrinsics as learnable parameters rather than taking them as input (Appendix A.6, p. 20).

14. **Intrinsics as output:** No. The core DELTA model does not output intrinsics; its output is per-pixel trajectories and visibility (Sec. 3, p. 3). In the appendix optimization pipeline (A.6, p. 20), per-keyframe intrinsics are optimized as learnable variables, but this is a downstream per-scene gradient-descent pipeline, not the core feedforward model.

15. **Moving cameras:** Yes. DELTA is designed for ordinary monocular videos where camera motion is present. The introduction explicitly names camera motion as one of the simultaneous challenges addressed (Introduction, p. 2). Appendix A.6 (p. 20) states that DELTA tracks "combine the camera motion from frame 0 to nu with any non-rigid scene motion." Evaluated on moving-camera datasets: TAP-Vid3D DriveTrack, Aria, PStudio (Tables 5, 7); LSFOdyssey (Table 4); in-the-wild videos (Fig. 5).

16. **Dynamic content:** Yes. Dynamic scenes and moving objects are the primary use case. The introduction identifies dynamic scene changes as a core challenge (Introduction, p. 2), qualitative results highlight moving foreground objects (Fig. 5, p. 9), and Appendix A.6 applies DELTA to dynamic video pose estimation and dynamic 3D scene reconstruction (pp. 20–22, Fig. 9).

17. **Feed-forward inference:** Yes. DELTA performs dense tracking in a single forward pass with no per-scene optimization. The paper explicitly contrasts it with per-video optimization methods (OmniMotion, etc.) and reports completing 100-frame dense tracking in under two minutes, over 8x faster than previous dense 3D methods (Abstract, p. 1; Introduction, pp. 2–3; Table 1, p. 2). The appendix pose estimation application adds a separate optimization stage, but this is explicitly a downstream application, not the core method.

**Notes:** DELTA requires an off-the-shelf monocular depth estimator as preprocessing — ZoeDepth or UniDepth for in-the-wild inference, synthetic Kubric depth for training. The model itself predicts scale-invariant log-depth change ratios, so 3D accuracy is tightly coupled to the depth estimator's quality and temporal scale-consistency (Sec. 3.4, p. 7; Sec. 5, p. 10; Appendix A.5, pp. 19–20). Training uses Kubric synthetic RGB-D videos with dense trajectory supervision; the training resolution is 384 x 512 with coarse tracking at 1/r^2 resolution followed by a transformer-based upsampler (Sec. 4.1, pp. 7–8; Appendix A.1, p. 16). A performance limitation is the fixed-window processing (best on videos under a few hundred frames) and potential failure on points occluded for extended periods (Sec. 5, p. 10). The appendix (A.6, pp. 20–21) demonstrates that DELTA tracks can serve as input to a gradient-descent pose/depth optimization pipeline competitive with visual odometry and SLAM methods, but this involves per-scene optimization and is not part of the core feedforward model.
